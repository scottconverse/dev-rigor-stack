dev-rigor-stack

A six-gate delivery discipline for AI coding agents — so "done" means proven, not "looks right."

View on GitHub Read the manual

What it is

AI coding agents are confident, and confidence isn't correctness. An agent says "done" when it means "I wrote code that looks right" — a different claim from "I proved it runs," different again from "I proved it's correct." This stack closes that gap: every change is exercised the way a real user hits it before anyone calls it finished, and the call to make a release real stays with a human. It ships as a set of Claude Code skills plus three always-on hooks — a reflex that primes every session, a rigor router that injects the matching task protocol per prompt, and a grounding check that blocks "done" on code nothing ever ran — and it exports to any other agent as plain text.

How it works

Two altitudes, one rule: nothing advances on a claim, only on evidence. Every unit of work — a fix, a feature, a refactor — walks a five-gate loop. Once per version, a heavier release gate runs over everything that merged, and ends at a human's go/no-go on the tag. An always-on layer primes every session, routes the right discipline to each prompt, and enforces a run-before-done floor; a continuity bookend keeps decisions from being lost across days and machines.

1 · PLANTrace the real code, write the test list and done-criteria, and classify blast radius — a one-line auth change is small in lines and large in blast, and gets the deep treatment.
2 · BUILD — /coder-tdd-qaTest-first. Write the failing test and watch it fail before making it pass; that, not a coverage number, is what proves the test is real.
3 · VERIFY — /proof-gateAdversarially try to break the claim. It survives only if it can't be refuted — killing checks that can't fail and "done" that outruns the evidence.
4 · REVIEWA review sized to the change: /audit-lite for a small diff, /audit-team for high blast, gauntletgate walkthrough for anything a user clicks.
5 · MERGEGreen-path only — a reviewed PR to main. No --admin, no override, no bypassing branch protection. One bad red merge undoes every green one after it.
Release gateOver the whole version: the full gauntlet driven to zero findings, the docs checked for overclaiming, a rollback plan named — then the owner's go/no-go on the tag.

The always-on layer

The six skills are pull-based — the agent invokes them when it judges a task needs them. Three hooks are push-based, each covering a different failure mode. The reflex is injected into every session and every subagent — a single page: the proof ladder, the never-shrink rules, a one-line evidence receipt. The rigor router reads each prompt and injects only the matching task protocol — bug work gets a reproduce-trace-root-cause investigation, visual and artifact work gets run-and-observe grounding, multi-part work gets decomposition with per-story evidence, release wording gets the release discipline; anything else gets silence, so context stays lean. And the grounding check is enforced, not suggested: if a session edited runnable code but never once executed or rendered anything, the agent is stopped and told to run the narrowest real check first. All three are convenience layers, not dependencies: the full discipline lives in the dev-rigor-stack skill.

Its core idea is proportion — spend exactly the rigor a change earns. Climb the ladder only as far as the blast radius demands, and stop at the first rung that covers it. No ceremony on a one-liner; no shortcuts on the auth path.

Some rules never shrink, no matter how small the change: watch a test fail before trusting it, never merge red or with --admin, never review your own work, never claim beyond the check you actually ran, and owner decisions (tag, publish, deploy, spend, delete) stay the owner's.

All six skills and the three hooks are MIT and authored by the repo owner — the installer installs all of them. audit-lite/audit-team and gauntletgate overlap by design: the same review discipline in two packagings — the standalone audits are the per-unit review reports, while gauntletgate is the release-altitude advancement gate (its lite/full lanes re-run that discipline self-contained, plus a pass/fail verdict). Get started

As Claude Code skills:

git clone https://github.com/scottconverse/dev-rigor-stack
cd dev-rigor-stack
./install.sh                                             # macOS / Linux / Git Bash
powershell -ExecutionPolicy Bypass -File .\install.ps1   # Windows

Requires Git and Node.js. The three hooks are small Node scripts — the six skills install without Node, only the hooks need it (and anyone running a coding agent almost certainly has it). On Windows the -ExecutionPolicy Bypass prefix avoids the default "scripts are disabled on this system" block.

Installs the six skills and wires the three always-on hooks into your settings.json; if Node is missing the skills still install and the installer says so. Re-running updates in place — idempotent, no path assumptions; a v1.4 install upgrades cleanly. ./install.sh --target ~/.codex/skills installs the skills elsewhere (e.g. Codex; no hooks).

Inside a Cowork or Codex session (no terminal): just ask the agent to install it from the repo — it copies the skills into place and, for a Claude install, wires the hooks.

For any other agent (ChatGPT, Gemini, Codex): run ./export/export-portable.sh and paste the generated portable-bundle.md into its system prompt. Claude-specific mechanics read as plain guidance — nothing is diluted from the Claude side to serve the export.

Honest scope

What it is

A repeatable discipline that makes an agent prove its work at the layer of the claim. Model-agnostic and tool-agnostic. A way to keep long, multi-machine efforts from losing their decisions.

What it isn't

Not magic, and not a guarantee of bug-free code. It's deliberate overhead that only pays off when being wrong is expensive — so it tells you, in every gate, when to skip it.
